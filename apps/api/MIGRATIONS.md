# Database Migrations

This project uses Alembic for database migrations with automatic migration support.

## Quick Start

Migrations run automatically when containers start. No manual action needed!

```bash
# Start services - migrations will auto-run
./dev.sh
```

## How It Works

### Automatic Migrations

The `entrypoint.sh` script automatically:
1. Waits for the database to be ready
2. Runs `alembic upgrade head` to apply all pending migrations
3. Starts the application

### Migration Files

Migration files are stored in `migrations/versions/` and follow the naming convention:
```
YYYYMMDD_HHMM_<revision>_<description>.py
```

## Creating New Migrations

When you modify models in `src/core/models.py`, create a migration:

```bash
# SSH into the API container
docker-compose exec api bash

# Create a new migration
./migrate.sh revision
# Enter a description when prompted, e.g., "add user preferences table"

# Or manually with alembic
alembic revision --autogenerate -m "add user preferences table"
```

### Review Before Applying

Always review the generated migration file before committing:

```bash
# Check what the migration will do
cat migrations/versions/20240115_1430_a1b2c3d4_add_user_preferences_table.py

# Apply it
alembic upgrade head
```

## Manual Migration Commands

```bash
# Run all pending migrations
./migrate.sh upgrade

# Rollback one migration
./migrate.sh downgrade

# Check current version
./migrate.sh current

# View migration history
./migrate.sh history

# Create empty migration (for manual SQL)
./migrate.sh revision
```

## Troubleshooting

### "Database not ready" errors

The entrypoint waits up to 30 seconds for the database. If it fails:

```bash
# Check database is running
docker-compose ps db

# Check database logs
docker-compose logs db

# Retry
docker-compose restart api
```

### "Migration failed" errors

If a migration fails:

```bash
# Check the error
docker-compose logs api

# Connect to database and check state
docker-compose exec db psql -U partymap -d partymap_bot

# Manual fix - stamp current version
alembic stamp head

# Or reset completely (DANGER - DATA LOSS)
docker-compose exec db psql -U partymap -d partymap_bot -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"
docker-compose restart api
```

### Conflicts with existing tables

If you're getting "table already exists" errors on a fresh install:

```bash
# Stamp the database as being at the latest version
alembic stamp head

# Then future migrations will work correctly
```

## Migration Best Practices

1. **Always review autogenerated migrations** - Check they match your intent
2. **Test migrations** - Run them on a copy of production data before deploying
3. **Make migrations reversible** - Always provide downgrade logic
4. **One change per migration** - Don't bundle unrelated changes
5. **Commit migration files** - They should be in version control
6. **Never modify existing migrations** - Create new ones to fix issues

## Schema Changes Guide

### Adding a Column

```python
# In models.py
new_field: Mapped[Optional[str]] = mapped_column(String(100), nullable=True)
```

```bash
./migrate.sh revision
# Migration will be auto-generated with ALTER TABLE ADD COLUMN
```

### Adding a Table

```python
# In models.py
class NewTable(Base):
    __tablename__ = "new_table"
    id: Mapped[UUID] = mapped_column(UUID(as_uuid=True), primary_key=True)
    # ... fields
```

```bash
./migrate.sh revision
# Migration will include CREATE TABLE
```

### Modifying a Column

Alembic doesn't auto-detect column type changes well. You may need to manually edit:

```python
# In the generated migration
def upgrade():
    op.alter_column('table_name', 'column_name', 
                    existing_type=sa.VARCHAR(length=50),
                    type_=sa.VARCHAR(length=100))
```

## Environment-Specific Notes

### Development
- Migrations auto-run on container start
- Source code mounted for hot reload
- Use `./migrate.sh` for manual commands

### Production
- Migrations run as part of deployment
- Always backup before running migrations
- Test migrations in staging first
- Consider maintenance mode for large migrations

## Migration Metadata

Alembic stores migration state in the `alembic_version` table:

```sql
SELECT * FROM alembic_version;
-- Returns: version_num
```

This tracks which migrations have been applied.
